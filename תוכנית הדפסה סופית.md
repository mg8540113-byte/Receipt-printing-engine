# תוכנית מלאה: מערכת הדפסת תלושים — Background Worker (גרסה סופית)

---

## 1. הקשר כללי

### המערכת הקיימת
אפליקציית Next.js על Vercel עם Supabase (PostgreSQL + Storage + Auth). המערכת מנהלת:

- **מוסדות** → בתוכם **קבוצות** → לכל קבוצה **הזמנות** (מאקסל) עם שם, ת.ז, פרטים, סכום
- לכל קבוצה מוגדרים **אחוזי חלוקה** שקובעים כמה תלושים מכל סוג (50/100/200) כל אברך מקבל
- **ברקודים** מגיעים ממקור חיצוני, נטענים למערכת בשלושה סוגים (50/100/200)
- **שלב הקצאה** — המערכת מחשבת לכל אברך כמה ברקודים מכל סוג, מקצה אותם לצמיתות, ושומרת על ההזמנה
- **כל זה כבר קיים ועובד בפרודקשן**

### מה בונים עכשיו
יכולת לייצר קבצי PDF פיזיים של תלושים — כ-150,000 תלושים, מחולקים לקבצים של עד 1,000 תלושים כל אחד. כל תלוש הוא עמוד אחד ב-PDF.

### הדרישה העסקית
המשתמש נכנס לעמוד ההדפסות באתר, בוחר מוסד → קבוצה, ולוחץ "הדפס". המערכת לוקחת את כל הנתונים שכבר קיימים (שם, ת.ז, ברקוד מוקצה, סוג תלוש, וכל השדות) ומייצרת PDF פיזיים לשליחה לדפוס מקצועי.

---

## 2. למה לא ישירות מ-Vercel

שלוש מגבלות קריטיות:

- **Timeout:** Vercel חותך פעולות אחרי 10-60 שניות (מקסימום 5 דקות). ייצור 1,000 תלושים לוקח יותר.
- **RAM:** מגבלת זיכרון לכל פונקציה. טעינת פונטים + תבניות + אלפי עמודים = קריסה.
- **אבטחה:** נדרש Service Role Key של Supabase. מסוכן לאחסן בפרויקט חשוף לאינטרנט.

## 3. ארכיטקטורת שלוש הצלעות

### צלע א' — Vercel (החזית / מנהל העבודה)

**תפקיד:** לקבל את בקשת ההדפסה מהמשתמש, לפצל אותה למשימות, להציג סטטוס ולינקי הורדה.

**מה הוא עושה:**
- המשתמש בוחר מוסד → קבוצה → לוחץ "הדפס"
- Vercel שולף את כל ה-vouchers המוקצים בקבוצה
- סופר כמה יש מכל סוג (50/100/200)
- מפצל לפי סוג תבנית, ובתוך כל סוג ל-batches של 1,000
- יוצר את כל שורות המשימה בטבלת print_jobs ב-**transaction אטומי אחד** (הכל או כלום)
- מחזיר למשתמש הודעה "העבודה החלה" — מיד, תוך שניות

**מה הוא לא עושה:** לא מייצר PDF, לא ניגש ל-Storage, לא מחזיק Service Role Key.

### צלע ב' — Supabase (המתווך והמחסן)

**תפקיד:** תור משימות + אחסון קבצים.

**מה הוא מחזיק:**
- טבלת print_jobs — רשימת כל המשימות עם סטטוס
- Storage bucket (print-files) — לאחסון קבצי ה-PDF הסופיים
- Realtime — שידור שינויים ל-Worker ולממשק

**מנגנון נעילה (Atomic Claim):** שימוש בשאילתת FOR UPDATE SKIP LOCKED כדי להבטיח שגם אם נרים כמה שרתים בעתיד, משימה לעולם לא תתבצע פעמיים.

### צלע ג' — Background Worker (הפועל)

**תפקיד:** ייצור ה-PDF בפועל.

**איפה הוא יושב:** שרת ב-Render.com, מוגדר כ-**Background Worker** (לא Web Service). זה קריטי — Background Worker ב-Render לא מקבל URL ציבורי. אין פורט פתוח, אין endpoint חשוף, אי אפשר לפנות אליו מבחוץ. הוא רק יוצא החוצה לדבר עם Supabase.

**עיקרון מפתח: ה-Worker הוא מדפסת טיפשה.** הוא לא מחשב כלום — לא סבסוד, לא חלוקה, לא הקצאה. כל הנתונים כבר מוכנים ב-DB. הוא רק שולף ושם על תבנית.

## 4. מבנה מסד הנתונים — טבלה חדשה

### טבלת print_jobs

| שדה | סוג | תיאור |
|-----|------|--------|
| id | UUID | מזהה ייחודי למשימה |
| batch_group_id | UUID | מזהה קבוצה — כל ה-batches שנוצרו מלחיצה אחת |
| group_id | UUID | מזהה הקבוצה (group) שממנה שולפים את הנתונים |
| template_type | INT | סוג תבנית: 50 / 100 / 200 |
| offset_start | INT | מאיזה תלוש להתחיל (למשל 0, 1000, 2000...) |
| limit_count | INT | כמה תלושים ב-batch הזה (עד 1000, האחרון יכול להיות קטן יותר) |
| status | ENUM | pending / in_progress / completed / failed |
| retry_count | INT | מספר ניסיונות חוזרים (מתחיל מ-0) |
| max_retries | INT | מקסימום ניסיונות (ברירת מחדל: 3) |
| worker_id | TEXT | מזהה ה-Worker שתפס את המשימה |
| started_at | TIMESTAMP | מתי התחיל העיבוד |
| completed_at | TIMESTAMP | מתי הסתיים |
| file_path | TEXT | נתיב הקובץ ב-Storage |
| pages_generated | INT | כמה עמודים נוצרו בפועל |
| error_message | TEXT | הודעת שגיאה אם נכשל |
| created_at | TIMESTAMP | מתי נוצרה המשימה |

אינדקסים: על status (ה-Worker שולף לפי pending), על batch_group_id (הממשק מציג סטטוס לפי קבוצה).

הטבלה **לא מתאפסת** אחרי הדפסה. משימות ישנות לא מפריעות לחדשות (ה-Worker שולף רק pending), והן שימושיות להיסטוריה, לינקי הורדה, ודיבאג. מחיקה תקופתית של שורות ישנות (60+ יום) תתבצע ביחד עם מחיקת קבצים מ-Storage.

### RPC: claim_print_job

פונקציית SQL שמקבלת worker_id ועושה UPDATE על שורה אחת שהסטטוס שלה pending, משנה ל-in_progress, שותלת started_at ו-worker_id, ומחזירה את השורה. השאילתה הפנימית חייבת להשתמש ב-FOR UPDATE SKIP LOCKED — מנגנון PostgreSQL שנועל את השורה ומדלג עליה אם כבר נעולה. מבטיח שמשימה נתפסת פעם אחת בלבד.

### RPC: reset_stuck_jobs

פונקציית Watchdog שעושה שני דברים: מוצאת משימות in_progress שתקועות יותר מ-10 דקות עם retry_count קטן מ-max_retries — מחזירה אותן ל-pending. מוצאת משימות שמיצו retry — מסמנת כ-failed.

### Storage Bucket

bucket בשם print-files. הגדרות: public (החלטה מודעת — המידע בתלושים אינו סודי, ויש דרישה עסקית לגישה ללא הגבלת זמן). שמות קבצים מבוססים על UUID.

### Realtime

הפעלת Realtime על טבלת print_jobs — ל-Worker (קבלת התראות על משימות חדשות) ולממשק (עדכוני סטטוס בזמן אמת).

## 5. זרימת התהליך המלאה

### שלב 1: דרישת עבודה (Vercel)
1. המשתמש בעמוד ההדפסות בוחר מוסד → קבוצה → לוחץ "הדפס"
2. ה-API Route מקבל group_id
3. Vercel שולף את כל ה-vouchers המוקצים בקבוצה הזו
4. סופר כמה מכל סוג: למשל 80,000 תלושי 50, 40,000 תלושי 100, 30,000 תלושי 200
5. מפצל כל סוג ל-batches של 1,000 (ה-batch האחרון יכול להיות קטן יותר)
6. מייצר batch_group_id אחד
7. **בתוך transaction אטומי אחד** — יוצר את כל שורות print_jobs
8. אם ה-transaction נכשל — שום שורה לא נכתבת, המשתמש מקבל שגיאה
9. אם הצליח — מחזיר batch_group_id, המשתמש רואה את מסך הסטטוס

### שלב 2: תפיסת משימה (Worker)
10. ה-Worker מקבל התראה דרך Realtime (או מגלה דרך polling כל 30 שניות)
11. מריץ את שאילתת Atomic Claim
12. אם קיבל משימה — מתחיל עיבוד
13. אם אין pending — חוזר להמתנה

### שלב 3: ייצור PDF (Worker)
14. ה-Worker שולף מ-Supabase את נתוני התלושים לפי group_id, template_type, offset_start ו-limit_count. השאילתה כוללת join לכל הנתונים הנדרשים: שם, ת.ז, טלפון, ברקוד, קוד מוסד, קוד הזמנה, תוקף
15. טוען מהזיכרון (כבר נטען בעליית השרת) את התבנית המתאימה ואת הפונט
16. **קורא את גודל העמוד מהתבנית** (220.8 × 110.8 מ"מ = 625.9 × 314.1 points) — לא מקודד מספרים
17. לכל תלוש: מוסיף עמוד בגודל התבנית → מחתים רקע (Page Embedding) → שותל טקסט עם bidi-js → שותל ברקוד עם bwip-js
18. אורז את ה-PDF

### שלב 4: ולידציה (Worker)
19. גודל קובץ > 0 בייט
20. מספר עמודים === limit_count של המשימה
21. גודל עמוד ראשון === גודל עמוד בתבנית המקורית
22. אם ולידציה נכשלת — המשימה מסומנת failed עם הודעה מפורטת

### שלב 5: העלאה ועדכון (Worker)
23. מעלה ל-Storage בנתיב jobs/{job_id}.pdf (idempotent — retry דורס קובץ ישן)
24. מעדכן print_jobs: status = completed, file_path, pages_generated, completed_at

### שלב 6: הצגה למשתמש (Vercel)
25. הממשק מאזין ל-Realtime ומציג progress bar בזמן אמת (X/150 הושלמו)
26. כל משימה שהושלמה — כפתור הורדה ירוק
27. כפתור "הורד הכל" — מוריד את כל הקבצים כ-ZIP אחד (JSZip בצד לקוח)
28. כפתור "מחק קבצים" — מוחק קבצים מ-Storage ומנקה file_path

## 6. טכנולוגיית ייצור PDF

### ספריית pdf-lib
עובדת ישירות על מבנה ה-PDF הבינארי. לא ממירה HTML, לא דורשת דפדפן, צורכת מעט זיכרון. PDF הוא וקטורי — דיוק מתמטי עד עשיריות הנקודה, ללא אובדן איכות. מתאים לדפוס מקצועי.

### Page Embedding
pdf-lib תומכת ב-embedPage שלוקחת עמוד מתבנית קיימת ויוצרת ממנו אובייקט משותף (XObject). כש-1,000 עמודים משתמשים בו, ה-PDF שומר רק הפנייה אחת — לא 1,000 עותקים. תוצאה: ~2MB ל-1,000 עמודים.

### גודל עמוד
**220.8 × 110.8 מ"מ (625.9 × 314.1 points).** גודל מיוחד, לא A4 ולא שום סטנדרט. ה-Worker קורא את הגודל מקובץ התבנית ולא מקודד מספרים. שלוש התבניות (50, 100, 200) באותו גודל בדיוק.

### ברקוד — bwip-js
מייצרת ברקוד Code 128 בזיכרון כ-PNG ב-**500 DPI**. מוטמע ישירות ב-PDF.

מפרט הברקוד:
- שטח לבן בתבנית: 1.5 × 3.0 ס"מ (לברקוד בלבד, הטקסט המספרי של 6 הספרות במקום אחר בתלוש)
- Quiet Zone: 0.3 ס"מ מכל צד
- ברקוד: 2.4 ס"מ רוחב, 1.2 ס"מ גובה
- רווח נשימה עליון ותחתון: 0.15 ס"מ כל אחד

## 7. טיפול בעברית (RTL)

### הבעיה
pdf-lib מציירת תווים משמאל לימין. "שלום" ייראה "םולש".

### הפתרון: bidi-js
ספריית bidi-js מממשת את ה-Unicode Bidirectional Algorithm (תקן UAX #9) — אותו אלגוריתם שדפדפנים ומערכות הפעלה משתמשים בו. מקבלת מחרוזת, מנתחת כל תו, מחזירה סדר וויזואלי נכון. מספרים נשארים LTR, עברית מתהפכת, סוגריים מתאימים נכון.

### למה לא split-reverse-join
שובר טקסט מעורב. "יוסי כהן (054)" הופך ל-")450( ןהכ יסוי". גם surrogate pairs (Unicode מורכב) נשברים.

### כלל
שדות עם עברית → bidi-js. שדות עם מספרים בלבד → ללא היפוך.

### הטמעת פונט
פונט עברי TTF מוטמע דרך @pdf-lib/fontkit. נטען פעם אחת לזיכרון בעליית ה-Worker.

## 8. מיפוי שדות התלוש

**מיקומי השדות זהים לשלוש התבניות** — טבלת קואורדינטות אחת בלבד. רק הרקע הגרפי שונה.

| שדה | סוג | טיפול |
|-----|------|--------|
| שם | עברית | bidi-js + פונט TTF |
| כולל | עברית | bidi-js + פונט TTF |
| ת.ז | מספרים | ללא היפוך |
| טלפון | מספרים | ללא היפוך |
| ברקוד (טקסט) | 6 ספרות | ללא היפוך, במיקום נפרד מתמונת הברקוד |
| ברקוד (תמונה) | PNG | bwip-js, Code 128, 500 DPI, עם Quiet Zone |
| תוקף | תאריך | ללא היפוך |
| קוד מוסד | 3 ספרות | ללא היפוך |
| קוד הזמנה | 4 ספרות | ללא היפוך |

המיקומים (x, y, fontSize) ייקבעו לפי מדידות שהמפתח יספק מתוך קובץ התבנית. יישמרו כאובייקט config אחד.

הערה: התלושים יוצאו בסדר שבו הם נשלפים מהמסד, ללא מיון.

## 9. מנגנוני חוסן (Resilience)

### Transaction על פיצול משימות
כל שורות print_jobs נכתבות ב-transaction אטומי אחד. הכל או כלום. מונע משימות חלקיות אם Vercel קורס באמצע.

### Atomic Claim
FOR UPDATE SKIP LOCKED. משימה נתפסת פעם אחת בלבד, גם עם כמה Workers.

### Polling Fallback
בנוסף ל-Realtime (trigger ראשי), ה-Worker שולח שאילתה כל 30 שניות — יש pending? רשת ביטחון למקרה שה-WebSocket נפל בשקט.

### Watchdog
כל 5 דקות, קריאה ל-RPC reset_stuck_jobs. משימות in_progress מעל 10 דקות חוזרות ל-pending.

### Retry עם מגבלה
retry_count עם max_retries=3. אחרי 3 ניסיונות — failed. מונע לולאה אינסופית.

### Idempotency
נתיב הקובץ: jobs/{job_id}.pdf. retry דורס קובץ ישן, לא יוצר כפיל.

### ולידציה לפני העלאה
גודל > 0, מספר עמודים = limit_count, גודל עמוד = גודל תבנית.

## 10. אבטחה

### Service Role Key
נשמר אך ורק כ-Environment Variable ב-Render. לא בקוד, לא ב-GitHub, לא ב-.env שנדחף ל-repo.

### ה-Worker סגור מהאינטרנט
מוגדר כ-Background Worker ב-Render. אין URL ציבורי, אין פורט פתוח. התקשורת היחידה: יוצאת ל-Supabase.

### אחסון קבצים
Storage bucket: public. החלטה מודעת — המידע אינו סודי, דרישה עסקית לגישה ללא הגבלה. שמות קבצים מבוססי UUID.

### ולידציית ברקודים
Unique Constraint ברמת PostgreSQL. ה-Worker סומך על שלמות הנתונים.

## 12. ממשק ההדפסה (בעמוד print הקיים)

### בחירה והפעלה
המשתמש בוחר מוסד → קבוצה (כמו ב-AllocateDialog הקיים) → לוחץ "הדפס תלושים". הדפסה כפולה מותרת — אותם ברקודים, אותם נתונים.

### תצוגת סטטוס בזמן אמת
Progress bar שמתעדכן דרך Realtime — X/150 הושלמו. רשימת משימות עם צבעים: אפור (ממתין), כחול/צהוב (בעבודה), ירוק (הושלם + הורדה), אדום (נכשל + שגיאה).

### הורדה
כפתור הורדה ירוק לכל משימה שהושלמה. כפתור "הורד הכל" שמוריד את כל קבצי ה-batch כ-ZIP אחד באמצעות JSZip בצד הלקוח.

### מחיקת קבצים
כפתור "מחק קבצים" לכל batch group. דיאלוג אישור → מוחק קבצים מ-Storage → מנקה file_path בטבלה. משחרר מקום ב-Storage.

## 15.5. תוספת — תמיכה בהזמנות חיצוניות (בוצע)

### רקע
נוסף תמיכה ב-print jobs מסוג "הזמנה חיצונית" — job שמגיע עם `external_order_id` ו-`group_id = null`.

### שינויים שבוצעו ב-Worker

**`src/db/queries.ts`**
- `PrintJob.group_id` הפך ל-`string | null`
- `PrintJob.external_order_id: string | null` נוסף
- `PrintJob.template_type` הוצמד ל-`50 | 100 | 200`
- `PrintJob.status` הוצמד ל-union type מלא
- `fetchVouchersByExternalOrder()` — פונקציה חדשה, קוראת ל-RPC החדש, מחזירה `VoucherRow[]` זהה

**`src/jobs/processor.ts`**
- Step 1 מנתב לפי `job.external_order_id` — if/else בין שתי פונקציות השליפה
- `jobMeta` כולל `external_order_id` ב-log

**`sql/fetch_vouchers_for_external_order.sql`** (קובץ חדש)
- RPC לסופהבייס — מחזיר את אותם 11 שדות של VoucherRow
- JOIN על `external_orders` ו-`barcodes`; ללא JOIN על orders/avrechim/groups/institutions
- ערכי ברירת מחדל: `NULL` לשדות שם נפרדים, `0` לavrech_code, literals לinstitution

### עיקרון
כל שאר הקוד (generator, validator, uploader, watchdog, retry) לא נוגע — VoucherRow זהה לחלוטין.

## 15. חלוקה ל-10 משימות

### צ'אט Cursor ראשון — פרויקט Vercel

**משימה 1 — Supabase Setup:** טבלת print_jobs, שני RPCs (claim + watchdog), Storage bucket, Realtime.

**משימה 2 — API Route:** endpoint שמקבל group_id, שולף vouchers, סופר לפי סוג, מפצל ל-batches, כותב ב-transaction.

**משימה 3 — ממשק הדפסה:** בחירת מוסד/קבוצה, כפתור הדפס, progress bar עם Realtime, כפתורי הורדה בודדים, כפתור "הורד הכל" (ZIP), כפתור מחיקת קבצים.

### צ'אט Cursor שני — פרויקט Worker

לפני פתיחת הצ'אט: להריץ שאילתה שמוציאה schema של טבלאות רלוונטיות (vouchers, barcodes, orders, avrechim, groups, institutions) ולהדביק כהקשר.

**משימה 4 — שלד:** אתחול repo, תלויות, TypeScript, חיבור Supabase עם Service Role Key. קובץ index שרק מתחבר ומדפיס "connected".

**משימה 5 — מנגנון תפיסה:** Realtime listener + polling כל 30 שניות + לולאת claim. בלי PDF — רק תופס משימה, מדפיס ללוג, מסמן completed.

**משימה 6 — Watchdog + retry:** קריאת reset_stuck_jobs כל 5 דקות + טיפול בשגיאות + logging מלא.

**משימה 7 — PDF בסיסי:** טעינת תבנית אחת + page embedding + שתילת טקסט אחד בלבד (בלי עברית, בלי ברקוד). קריאת גודל עמוד מהתבנית. ולידציה שהקובץ נוצר.

**משימה 8 — עברית:** fontkit + bidi-js + שתילת כל שדות הטקסט במיקומים מדויקים.

**משימה 9 — ברקוד:** bwip-js + Code 128 + 500 DPI + שתילה בשטח 1.5×3.0 ס"מ עם Quiet Zones.

**משימה 10 — ולידציה + העלאה + 3 תבניות:** בדיקת גודל/עמודים/מידות, העלאה ל-Storage, עדכון סטטוס, תמיכה בשלוש תבניות.

### סדר ביצוע
קודם צ'אט 1 (משימות 1-3) כי ה-Worker תלוי בטבלה וב-RPCs. אחר כך צ'אט 2 (משימות 4-10) בסדר — כל משימה בונה על הקודמת.
